## Короткий вывод

Сделан не демо-скрипт "один prompt в LLM", а контролируемый пайплайн:

- транскрипт режется на проверяемые anchors;
- LLM работает по ограниченным фрагментам;
- ответ жёстко ограничен JSON Schema;
- каждая задача должна иметь evidence из исходного текста;
- даты нормализуются кодом, а не доверием к модели;
- финальный результат строится через 5 прогонов и consensus;
- служебные метрики сохраняются в SQLite;
- итоговый `DataFrame` остаётся в формате задания.

По галлюцинациям результат уже рабочий: текущий precision против golden-разметки
`91.3%`. По воспроизводимости цель ещё не достигнута: текущая строгая
повторяемость задач на уровне `5/5` составляет `57.5%`, а нужно около `90%`.

## Что требовалось получить

Итоговый объект должен быть `pandas.DataFrame` с колонками:

```text
Блок | Задача | Ответственный | Срок | Обоснование
```

Дополнительные требования, которые мы взяли как инженерный критерий качества:

- любой человек должен глазами сверить задачу с `Обоснование`;
- неподтверждённый evidence нельзя тихо заменять текстом anchor;
- 5 повторных прогонов должны давать примерно один и тот же результат;
- финальный результат не должен зависеть от последнего случайного run;
- SQLite как дополнительное задание должен хранить не только финальные строки,
  но и audit-слой по каждому прогону.

## Выполненный объём работ

### Архитектура и API-ресурсы

- Сделан CLI в `main.py` для одного файла, всех файлов, golden eval, SQLite и
  lexicon audit.
- Добавлен режим `--strategy anchored` по умолчанию.
- Сохранены старые режимы `grouped` и `global` для сравнения.
- Добавлены `--runs`, `--min-consensus-share`, `--max-stability-delta`,
  `--verify-near-consensus`, `--verifier-min-support-share`.
- LLM получает не весь транскрипт, а anchors с перекрытием.
- Вход перед API сокращается на 60.6-82.9% в зависимости от файла.

### Preprocess и anchors

- Парсинг Zoom-транскрипта в реплики speaker/text.
- High-recall anchors по task intent, срокам, done/failed сигналам и recap.
- Date-first anchors: строка со сроком попадает в контекст даже без слова
  "задача".
- Final-tail fallback для низкого coverage.
- Merge пересекающихся anchors.
- Узкие окна для done/failed, чтобы не подтягивать соседние будущие действия.
- Clause-level status detection, чтобы отрицание в одной части фразы не
  убивало done-сигнал в другой.

### Даты

- Общие date patterns вынесены в `src/date_patterns.py`.
- Нормализуются `сегодня`, `завтра`, `послезавтра`, дни недели, `через неделю`,
  `на этой неделе`, `на следующей неделе, в среду`, `к следующей пятнице`,
  `в конце недели`, `до конца недели`.
- Для `Новые` срок берётся только из evidence.
- Для `Выполненные` срок очищается всегда.
- Добавлены границы слов, чтобы `завтра` не матчился внутри `послезавтра` или
  `завтрашний`.

### Postprocess и защита от галлюцинаций

- Evidence должен быть подтверждён исходным anchor.
- Неподтверждённый evidence не заменяется автоматически.
- `deadline_raw` от LLM не считается доказательством срока для новых задач.
- Выполненные задачи проверяются на done-сигнал в той же clause, где объект
  задачи.
- Отрицания вроде `не сделали`, `не готова`, `не выполнено`, `не закрыты` не
  проходят как `Выполненные`.
- Ongoing/current state вроде `уже проводим`, `производится`, `сейчас в работе`
  отфильтровывается из `Новые`, если нет явного assignment.
- Dedupe выполняется после validation, чтобы галлюцинация не вытеснила валидную
  короткую цитату.

### Ответственные

- Исправлено смешение `ответственный` и `ответственность`.
- Поддержаны конструкции вроде `ответственным будет Петров`.
- Для merged multi-speaker anchors fallback идёт от speaker в evidence, а не от
  первой реплики anchor.
- Recap-парсинг ответственного ограничен имя-подобными токенами.
- Убраны наиболее опасные org-tail ошибки в духе `Кудинова Югра`.

### Stability, consensus и verifier

- Финальный DataFrame больше не берётся из последнего прогона.
- Считаются counts по блокам, one-off rows, Jaccard, centroid-run,
  postprocess stats.
- Финальный strict consensus требует `support>=5` при 5 прогонах и пороге 90%.
- Near-consensus verifier проверяет кандидатов с поддержкой 3/5 или 4/5.
- В `StabilityReport` хранятся `support_count`, `support_ratio`, `run_indices`,
  `verification_status`.
- Эти поля не попадают в итоговый DataFrame.

## Baseline и сравнение подходов

### Исторический grouped baseline

В Obsidian и `runs/llm_grouped_full_after_merge.txt` зафиксирован ранний
режим `grouped` с мягким `consensus support>=2`.

Он давал больше строк, но был слишком свободным:

| Файл | Consensus support>=2 | Проблема |
| --- | ---: | --- |
| `transcript.txt` | 22 из 28 | много weak done и verbose duplicates |
| `transcript2.txt` | 4 из 4 | лучший стабильный файл |
| `transcript3.txt` | 14 из 18 | были дубли и ongoing new tasks |

Этот baseline полезен как история разработки, но он не соответствует целевому
критерию `90%`: строка могла попасть в финал уже при поддержке 2/5.

### Строгие 90%-режимы из SQLite

| Режим | Precision | Recall | Strict 5/5 воспроизводимость | Средний min Jaccard | Файлов прошли 90% |
| --- | ---: | ---: | ---: | ---: | ---: |
| `tasks_global.sqlite` | 100.0% | 45.5% | 15/42 = 35.7% | 37.8% | 0/3 |
| `tasks_anchored.sqlite` | 95.7% | 66.7% | 23/35 = 65.7% | 76.5% | 1/3 |
| `tasks_verified.sqlite` | 91.3% | 63.6% | 23/40 = 57.5% | 65.5% | 0/3 |

Интерпретация:

- `global` слишком нестабилен и теряет recall.
- `anchored` сильно улучшил recall и стабильность.
- `verifier` в последнем прогоне добавил одну полезную near-consensus строку,
  но не улучшил общую стабильность. Его нужно оставить как audit-инструмент,
  а не считать решением проблемы variance.

## Последний результат: `tasks_verified.sqlite`

### Стабильность по файлам

| Файл | Финальных строк | Strict 5/5 | Воспроизводимость | Min pairwise Jaccard | Max count delta | 90% пройден |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| `transcript.txt` | 9 | 9 из 21 | 42.9% | 50.0% | 100% | Нет |
| `transcript2.txt` | 3 | 3 из 4 | 75.0% | 75.0% | 100% | Нет |
| `transcript3.txt` | 11 | 10 из 15 raw, +1 verifier | 73.3% с verifier | 71.4% | 50% | Нет |

Почему max count delta может быть высоким при неплохом результате: метрика
считает относительную разницу по блокам. Если в одном run блок `Новые` равен 0,
а в другом 1, относительное расхождение становится 100%, хотя абсолютная
разница маленькая.

### Качество против golden-разметки

| Файл | Expected | Predicted | Matched | Precision | Recall |
| --- | ---: | ---: | ---: | ---: | ---: |
| `transcript.txt` | 16 | 9 | 8 | 88.9% | 50.0% |
| `transcript2.txt` | 4 | 3 | 3 | 100.0% | 75.0% |
| `transcript3.txt` | 13 | 11 | 10 | 90.9% | 76.9% |
| Итого | 33 | 23 | 21 | 91.3% | 63.6% |

Полное совпадение по ответственному и сроку у найденных golden-строк:
`21/21`. То есть когда задача найдена и сматчена, поля сейчас в основном
заполняются корректно. Главная потеря качества - пропуски.

### Что пропущено и что шумит

`transcript.txt`:

- пропущены: чек-листы до 15 апреля, шаблон типового слайда, все мероприятия
  по КМ, рассылка формы, запрос владельцам процессов, встреча с Чайка,

`transcript2.txt`:

- пропущено: `через неделю собраться и ещё раз обсудить`;
- false positives нет.

`transcript3.txt`:

- пропущены: `подготовил машину`, `завтра обсудить с Леонидом`,

- false positive: `Я просто не успел`.

## Что получилось хорошо

1. Формат DataFrame приведён к ТЗ и защищён тестами.
2. SQLite хранит полный audit trail, а не только финальную таблицу.
3. Evidence теперь проверяемый: строку можно глазами сверить с транскриптом.
4. Ответственные и сроки перестали массово переноситься из соседних реплик.
5. `anchored` заметно лучше `global`: recall вырос с 45.5% до 66.7%, средний
   min Jaccard с 37.8% до 76.5% в лучшем сохранённом strict-прогоне.
6. Тестовая база выросла до 90 тестов и закрывает регрессии по датам,
   статусам, responsible, consensus и SQLite.

## Что пока не получилось

1. Целевые `90%` воспроизводимости ещё не достигнуты.
2. Строгий `support>=5` режет реальные задачи, которые модель находит в 3-4
   прогонах.
3. Verifier не решил variance. Он скорее полезен для аудита near-consensus,
   чем для стабилизации extraction.
4. `Выполненные` остаются самым сложным блоком: done-сигналы часто похожи на
   описание текущего состояния или общий комментарий.
5. Новые задачи в финальных recap-блоках могут выпадать, если LLM по-разному
   формулирует одну и ту же строку.

## Дальнейшие улучшения

### 1. Deterministic candidate registry

Сейчас каждый run сам формулирует список задач. Это создаёт variance.

Следующий шаг: до LLM создать стабильный реестр кандидатов:

- candidate id;
- anchor ids;
- evidence span;
- candidate kind;
- найденные date phrases;
- speakers;
- локальные маркеры.

LLM должна не генерировать свободный список, а принимать решение по уже
заданным кандидатам. Это должно сильнее всего повысить 5/5 воспроизводимость.

### 2. Канонизация задач до consensus

Нужно нормализовать формулировки до сравнения:

- убрать порядок слов;
- привести `отчёт/отчет`, `бизнес-процесс/бизнеспроцесс`;
- лемматизировать ключевые action/object tokens;
- сравнивать не весь текст, а `action + object + deadline + evidence span`.

Это можно сделать без тяжёлой ML-модели, а E5 оставить как optional fallback.

### 3. Connected anchor groups

Фиксированные пачки anchors всё ещё могут разрезать связанные задачи.

Лучше группировать anchors по:

- близости line ids;
- общим date phrases;
- общим speakers;
- пересечению signal windows;
- принадлежности к финальному recap.

### 4. Verifier как field-level audit

Verifier стоит перестроить: пусть он не "спасает" задачу целиком, а проверяет
отдельно:

- подтверждён ли block;
- подтверждена ли task;
- подтверждён ли responsible;
- подтверждён ли deadline;
- есть ли риск галлюцинации.

Так его решение станет объяснимым.

### 5. Улучшить recall без роста галлюцинаций

Для пропусков нужны не новые случайные regex, а проверяемый процесс:

1. Добавить missed golden case в regression fixture.
2. Проверить, попал ли фрагмент в anchors.
3. Если не попал, расширить preprocess.
4. Если попал, но LLM не взяла, ужесточить candidate contract.
5. Если LLM взяла нестабильно, улучшить canonical consensus.

### 6. После этого решать по E5

E5/SentenceTransformers может помочь в similarity, но сейчас не он главный
блокер. Сначала нужен стабильный candidate registry. Потом можно проверить,
даёт ли E5 прирост в:

- dedupe;
- task similarity;
- evidence support;
- grouping consensus.
